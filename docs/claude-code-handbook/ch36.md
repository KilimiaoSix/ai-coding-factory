# 第 36 章：插件系统——第三方扩展机制

## 问题定义

如果技能是提示词层扩展，那么插件就是更完整的第三方能力包。它们可以带来命令、MCP 服务器、hooks、agents、output styles 等多种组件，需要一套独立的安装、校验、启停和策略控制系统。

## 架构分析

当前快照的插件体系分成三层：`src/services/plugins/` 提供 CLI 操作封装与安装管理；`src/utils/plugins/` 负责加载、校验、市场源、依赖与本地目录处理；`src/plugins/` 和 `builtinPlugins.ts` 负责内置插件注册。插件可以以用户、项目、本地或 session 级别生效，并与策略、遥测和 Marketplace 元数据联动。

## 关键源码锚点

- `src/services/plugins/pluginCliCommands.ts`
- `src/services/plugins/pluginOperations.ts`
- `src/utils/plugins/pluginLoader.ts`
- `src/utils/plugins/validatePlugin.ts`
- `src/utils/plugins/marketplaceManager.ts`
- `src/plugins/builtinPlugins.ts`

## 快照修正与补充

- `other-ans/ch36.md` 将插件描述为“第三方扩展机制”，而当前快照已经能看到完整的 install/list/validate/update 管线。
- 内置插件与 bundled skills 不是一回事，`builtinPlugins.ts` 明确说明了二者差异：插件可出现在 `/plugin` UI 中并可被启停。
- 外部快照里插件功能比多 Agent 和 Bridge 更完整，但仍会受到策略、来源和配置限制。

## 设计启示

- 插件系统要成功，核心不是“能加载”，而是“能验证、能治理、能停用”。
- 让插件进入统一命令、MCP、hook 与 agent 入口，能避免形成平行扩展体系。
- 安全边界和来源管理应当是插件系统的默认部分，而不是后加。
