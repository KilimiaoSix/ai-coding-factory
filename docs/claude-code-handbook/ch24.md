# 第 24 章：Swarm——多 Agent 协调架构

## 问题定义

当任务复杂到单一上下文无法覆盖时，系统需要的不只是“再开一个 Agent”，而是协调多个 worker、同步结果、保持权限边界的协作模型。Swarm 就是这种协作模式的抽象。

## 架构分析

当前快照的多 Agent 组织主要围绕 coordinator mode、AgentTool 和 task 系统展开。协调者负责分解任务和汇总结果，worker 以本地、远程或 worktree 隔离方式执行。团队状态、消息通知和能力限制则通过 agent/task 辅助模块维持。

## 快照修正与补充

- `other-ans/ch24.md` 对 Team File、邮箱系统和 backend 模式有更完整的体系化叙事，但在当前外部快照里，并非所有子系统都以同名公开目录存在。
- 可以可靠确认的事实是：协调者模式、worker 可用工具集合、任务通知格式和运行时状态管理都在源码里有入口。
- `COORDINATOR_MODE` 明确受 feature gate 控制，默认外部构建并不会把它作为稳定可用主路径。

## 设计启示

- 多 Agent 系统最难的不是“并发”，而是“谁负责协调、谁负责收口”。
- 协调者必须有受控的 worker 能力集合，否则多 Agent 只会放大风险。
- Agent 协作需要显式的任务和通知协议，而不能靠隐式共享上下文。
