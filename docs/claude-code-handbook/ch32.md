# 第 32 章：消息渲染——从 AI 输出到终端显示

## 问题定义

模型输出不是简单的纯文本。Claude Code 需要把助手文本、thinking、tool use、tool result、系统消息、附件、拒绝信息和计划审批都渲染成不同的终端组件。

## 架构分析

当前快照中的消息渲染被拆成两层：上层是 `Messages` 与 `src/components/messages/` 下的大量消息组件，下层是 Ink 文本与布局系统。这样做的好处是每类消息都能独立演进，例如 read/search 内容可以折叠，工具结果可以按成功/拒绝/错误分类，plan approval 可以单独占据交互区域。

## 快照修正与补充

- `other-ans/ch32.md` 强调“从 AI 输出到终端显示”的消息渲染管线，这和当前 `components/messages` 的细粒度拆分完全对应。
- `docs/08-ui-components.md` 只讲宿主框架，本章补上消息层的产品语义。
- thinking、tool use、compact boundary 和任务通知都已经成为一等消息类型，而不是调试输出。

## 设计启示

- 对 Agent UI 来说，消息结构必须比聊天应用更丰富。
- 把不同消息类型拆成组件，是控制复杂度和强化可观察性的必要条件。
- 消息渲染越结构化，用户越容易理解模型当前在做什么。
