# 第 44 章：应用状态机——从引导到就绪

## 问题定义

启动流程描述的是步骤顺序，应用状态机描述的是“系统当前处在什么状态、能否进入下一阶段”。对 Claude Code 来说，这包括配置是否启用、信任是否建立、远程设置是否加载、权限上下文是否准备好、REPL 是否可交互。

## 架构分析

当前快照没有把所有生命周期状态画成单一枚举，而是通过 `bootstrap/state.ts`、`AppState`、初始化 promise、权限上下文和 REPL 条件渲染共同形成状态机。你可以把它理解为多个子状态机叠加：启动子状态机、认证子状态机、权限子状态机、会话子状态机和 UI 子状态机。

## 快照修正与补充

- `other-ans/ch44.md` 用“应用状态机”概括生命周期，是一种很合适的阅读框架；当前快照虽然分散实现，但状态边界确实存在。
- `docs/00-architecture-overview.md` 与 `docs/09-config-and-state.md` 分别给出架构和状态层视角，本章把二者放到“从未就绪到就绪”的时间轴上。
- 用户是否接受 trust dialog、远程设置是否加载、GrowthBook 是否刷新，都会改变系统后续路径。

## 设计启示

- 大型 CLI/Agent 应用并不是“启动完就完事”，而是持续处于多个生命周期子状态中。
- 把生命周期拆成子状态机，比维护一个庞大的中央状态枚举更实用。
- 文档化状态机，能帮助理解很多“为什么这个功能此时不可用”的问题。
