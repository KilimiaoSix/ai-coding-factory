# 第 3 章：核心设计原则

## 问题定义

`other-ans/ch03.md` 把 Claude Code 总结成七条原则：流式优先、默认隔离、权限最小化、渐进增强、用户始终可控、优雅降级、可观测性。本章将这些原则对应到快照里的真实代码。

## 架构分析

- 流式优先：`src/query.ts`、`src/services/api/claude.ts` 和 `src/ink/` 都围绕“边到达边消费”设计。
- 默认隔离：`Permission Mode`、worktree、沙箱 UI、session 目录、动态技能加载都优先限制边界。
- 权限最小化：Tool 必须显式声明读写属性和权限检查，见 `src/Tool.ts` 与 `src/utils/permissions/`。
- 渐进增强：大量能力通过 `feature()`、GrowthBook、环境变量和插件机制启用，而不是硬编码到主路径。
- 用户始终可控：用户可以中断、确认、切换模式、切换模型、修改命令和权限规则。
- 优雅降级：MCP 失败、压缩失败、模型回退、内部包缺失时，系统尽量保持主流程可继续。
- 可观测性：Analytics sink、会话 JSONL、diagnostics、startup profiler 都在追踪运行状态。

## 关键源码锚点

- `src/query.ts`
- `src/Tool.ts`
- `src/utils/permissions/`
- `src/services/analytics/index.ts`
- `src/utils/sessionStorage.ts`
- `scripts/build.ts`

## 快照修正与补充

- `docs/00-architecture-overview.md` 与 `other-ans` 对原则的总结基本一致，但源码快照最能说明问题的地方往往不是“主类”，而是边角模块，例如 `sessionStorage`、`analytics`、`keychainPrefetch`。
- 渐进增强在当前仓库中尤为重要，因为外部快照明确把很多内部能力通过 `scripts/build.ts` 置为关闭或桩实现。
- 优雅降级不是抽象口号，而是外部快照得以正常构建的前提。

## 设计启示

- 在 Agent 项目里，原则不是补充说明，而是决定权限、工具、上下文、UI 这些模块怎么写的源头。
- 如果没有“默认隔离 + 优雅降级”，终端 Agent 很容易从“强大”滑向“不稳定或不安全”。
- 将原则落实到文件和流程，比只停留在宣言更有价值。
