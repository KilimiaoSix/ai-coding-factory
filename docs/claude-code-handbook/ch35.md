# 第 35 章：命令系统——可插拔的命令架构

## 问题定义

Claude Code 既是 Agent，也是 CLI。除了模型驱动的 Tool，它还要支持大量用户显式命令，如 `/compact`、`/config`、`/permissions`、`/teleport`、`/plugin` 等，因此命令系统必须足够可扩展。

## 架构分析

`src/commands.ts` 把命令统一注册为 `Command` 集合，并根据 `type` 区分 prompt、local、local-jsx 等不同执行方式。命令来源不仅有内置命令，还包括技能目录、bundled skills、MCP skills、workflow 命令和 feature-gated 扩展。

## 关键源码锚点

- `src/commands.ts`
- `src/types/command.ts`
- `src/commands/`
- `src/skills/loadSkillsDir.ts`
- `src/skills/bundledSkills.ts`
- `src/services/mcp/client.ts`

## 快照修正与补充

- `docs/05-command-system.md` 已给出命令类型定义，本章强调“可插拔来源”与“命令注册入口”的整体意义。
- `other-ans/ch35.md` 把命令系统视为扩展性支点，这和当前 `getCommands()` 实际承接技能、MCP 与工作流命令的实现一致。
- 命令与 Tool 虽然都能触发行为，但前者由用户显式调用，后者由模型在一次对话中调度，两者的交互边界必须清楚。

## 设计启示

- 命令系统不应只是一个巨型 switch，而应支持多来源注册和延迟加载。
- 将“用户主动触发能力”与“模型自动调用能力”分离，是终端 Agent 体验的关键。
- 命令架构越清晰，产品越容易在不改主循环的前提下持续扩展。
