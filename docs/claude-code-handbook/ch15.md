# 第 15 章：文件操作工具的设计哲学

## 问题定义

对编程 Agent 来说，文件读写是最常见也最容易出问题的能力。好的文件工具不仅要“能改”，还要能限制破坏面、保留上下文、与权限系统协作。

## 架构分析

当前快照里的文件工具并不是一个统一的“filesystem API”，而是拆成了 `FileReadTool`、`FileEditTool`、`FileWriteTool`、`NotebookEditTool` 等不同能力，每个工具自行声明输入格式、只读属性和权限检查。这样做的优点是：读取、精确编辑、整文件覆盖和 notebook 修改可以分别约束，而不是混在一个过于强大的超级工具里。

## 快照修正与补充

- `docs/03-tool-system.md` 提到了文件工具，但没有展开“为什么拆成多个 Tool”；这正是本章补充的重点。
- `other-ans/ch15.md` 强调精确替换优于模糊行号修改，这与当前快照中 `Edit` 工具强调结构化输入是一致的。
- 文件工具还与 sessionStorage、fileHistory、权限建议和结构化 diff UI 产生联动，这些跨模块收益往往比“写文件”本身更重要。

## 设计启示

- 把不同写入语义拆成多个工具，比做一个全能写文件工具更安全。
- 文件读取能力不应被简单视为无风险，因为它仍会影响上下文大小和敏感信息暴露。
- 对编程 Agent 来说，文件工具的真正价值在于“可控修改”，而不是“任意修改”。
