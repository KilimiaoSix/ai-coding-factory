# 第 29 章：状态变更与响应式更新

## 问题定义

有了全局状态模型之后，真正决定体验的是：状态变化如何驱动 UI 重绘、权限上下文刷新和副作用执行，而不会导致整个终端界面抖动或失去一致性。

## 架构分析

当前快照通过 `setState` 的不可变更新、selector、`useSyncExternalStore` 和 `onChangeAppState` 等机制把状态变化控制在可预测范围内。React 负责组件级订阅，store 负责发布变化，辅助模块再把状态变化转化为日志、设置同步或任务联动。

## 快照修正与补充

- `other-ans/ch29.md` 提到 selector 模式与副作用监听，这在当前快照的 state 目录中是可以直接验证的。
- `docs/09-config-and-state.md` 中“响应式状态管理”的描述比较概括，本章把它展开到实际的订阅与变更传播机制。
- 设置变化与权限变化不是直接写 UI，而是先更新状态，再由组件树响应。

## 设计启示

- 状态变化要尽量保持单向传播，否则终端渲染与工具执行容易互相打架。
- selector 和局部订阅是终端 UI 性能的必要条件。
- 副作用应挂在状态边界上，而不是散落在业务逻辑内部。
