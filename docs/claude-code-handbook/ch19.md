# 第 19 章：MCP——开放的工具协议

## 问题定义

内置工具只能覆盖有限能力。MCP 的作用是让 Claude Code 能在统一 Tool 框架里接入外部服务器，把第三方能力包装成与内置工具等价的可调用对象。

## 架构分析

`src/services/mcp/client.ts` 是 MCP 能力中心：负责创建 stdio、SSE、HTTP、WebSocket 等传输，建立会话，列出工具/资源/提示，处理鉴权、输出截断、二进制持久化和错误翻译。其上层再通过 `MCPTool`、`ListMcpResourcesTool`、`ReadMcpResourceTool` 等统一暴露给模型和用户界面。

## 关键源码锚点

- `src/services/mcp/client.ts`
- `src/services/mcp/MCPConnectionManager.tsx`
- `src/services/mcp/InProcessTransport.ts`
- `src/tools/MCPTool/MCPTool.ts`
- `src/tools/ListMcpResourcesTool/ListMcpResourcesTool.ts`
- `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts`

## 快照修正与补充

- `docs/02-service-layer.md` 把 MCP 列为服务层组成部分，但实际 `client.ts` 的职责非常宽，既处理连接，也处理工具暴露与资源读取。
- `other-ans/ch19.md` 强调传输层抽象和无缝融入工具体系，这在当前快照中完全成立。
- 某些 MCP 相关特性，例如 MCP skills 或更深入的鉴权路径，仍然受 feature gate 影响。

## 设计启示

- 外部协议要想真正融入 Agent，必须转换成统一 Tool 契约，而不是停留在“客户端 SDK”层面。
- MCP 最难的部分往往不是连接，而是把远程输出压缩成适合上下文窗口的内容。
- 开放扩展与权限控制必须一起设计。
