# 第 7 章：工具调度与并发执行

## 问题定义

模型可以在一轮中生成多个 tool call。系统并不是简单“按顺序跑完”，而是要判断哪些调用可以并发、哪些必须串行、哪些错误需要级联终止。

## 架构分析

当前快照保留了两条执行路径：流式阶段的 `StreamingToolExecutor`，以及回合内收尾阶段的 `runTools`。Tool 接口中的 `isConcurrencySafe`、`isReadOnly`、`isDestructive` 等声明，为调度器提供了安全分类依据。只读搜索类工具更适合提前并发启动，写操作和高风险工具则偏向串行或严格受控。

## 快照修正与补充

- `docs/03-tool-system.md` 主要解释 Tool 接口，本章则关注“多个 Tool 在一次模型回复里如何被排队和调度”。
- `other-ans/ch07.md` 强调错误级联与中断语义；这与当前快照在 `AbortController`、Bash 错误处理和工具队列中的实现方向一致。
- 并发不是默认策略，而是声明式白名单策略。

## 设计启示

- 对 Agent 来说，并发调度首先是安全问题，其次才是性能问题。
- 工具元数据足够丰富，调度器才能通用化。
- “可以并发”最好由工具自己声明，而不是由调度器猜。
