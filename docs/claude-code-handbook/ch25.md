# 第 25 章：Worktree——并行工作流

## 问题定义

多 Agent 并行修改同一仓库时，最现实的隔离方案不是复制整仓，而是利用 Git worktree。Claude Code 的 worktree 支持围绕这一点建立：给子 Agent 一份隔离的工作目录，同时尽量复用大体量依赖。

## 架构分析

`src/utils/worktree.ts` 负责 worktree 名称校验、Git worktree 创建与删除、目录准备、依赖目录软链接、设置文件同步和 hook 执行。对 AgentTool 来说，worktree 是一种隔离模式；对用户来说，它则是一种能并行推进任务、同时避免主工作区被污染的工程手段。

## 快照修正与补充

- `other-ans/ch25.md` 对 worktree 的并行开发叙事很强，当前快照中的 `validateWorktreeSlug`、symlink 逻辑和 hook 调用说明这确实是工程化实现，而不是概念占位。
- `docs/09-config-and-state.md` 提到 `worktreeSessions` 持久化字段，这意味着 worktree 也进入了项目级配置状态。
- 并不是所有 agent 都必须跑在 worktree 中，它是隔离策略之一，而非唯一模式。

## 设计启示

- 并行工作流最有效的隔离往往来自 Git 语义，而不是额外造一套文件复制系统。
- worktree 名称与路径校验本身就是安全问题。
- 能复用 `node_modules` 等重资源目录，决定了 worktree 模式能否在真实项目里被频繁使用。
