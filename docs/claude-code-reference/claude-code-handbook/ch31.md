# 第 31 章：React 在终端——Ink 框架的设计

## 问题定义

Claude Code 选择 React + Ink 作为终端 UI 基础，不是因为“终端也想套前端框架”，而是因为这个产品的界面已经具备复杂应用特征：流式消息、输入框、弹窗、任务列表、权限浮层、搜索高亮和虚拟滚动都需要统一的声明式状态驱动。

## 架构分析

当前快照维护了一个深度定制的 Ink 分支：React Reconciler 把组件树映射成自定义 DOM，再由 Yoga 参与布局计算，最终渲染到终端 screen buffer。Claude Code 在这个基础上增加了 ScrollBox、文本选择、搜索高亮、点击事件和 alternate screen 等终端特性。

## 设计启示

- 终端 UI 一旦需要承载长会话与复杂状态，声明式渲染的收益会迅速超过命令式输出。
- 宿主环境做得越完整，越容易把“终端”当成真正的应用界面，而不是日志窗口。
- 对 Agent 产品而言，UI 架构会直接影响推理过程是否可见、可打断、可审计。
