# 第 27 章：后台任务与异步执行

## 问题定义

并非所有任务都适合阻塞在当前对话前台。长时间命令、子 Agent、远程执行和监控类任务需要进入后台，系统再用通知与状态面板把结果带回当前会话。

## 架构分析

任务系统把不同执行体统一建模成 `TaskState`：本地 Shell、LocalAgent、RemoteAgent、InProcessTeammate 等都进入同一状态集合，再由 UI 和工具层消费。任务工具负责创建、查询、更新、停止任务，任务文件和输出文件则承接结果持久化。

## 快照修正与补充

- `other-ans/ch27.md` 把后台任务视为 Agent 自动化的重要一步，这一判断与当前 task 子系统高度吻合。
- 当前外部快照包含多类 task 状态和 UI 组件，但一些具体 backend 能力仍然可能受门控影响。
- 后台任务与 AgentTool 关系紧密，但并不等价；前者是通用执行容器，后者是一个具体的“启动子 Agent”的能力。

## 设计启示

- 任务系统是把“对话式 Agent”推向“长期运行执行体”的关键桥梁。
- 如果后台任务没有统一状态模型，UI 和权限系统很快就会失控。
- 长任务必须是可停止、可查询、可恢复的。
