# 第 26 章：Teleport——远程协作

## 问题定义

Teleport 解决的是“把本地会话继续到另一台机器或远程环境中”的问题。相比普通 resume，它要同时处理远程 API、Git 源、日志回放、分支恢复和 OAuth 鉴权。

## 架构分析

当前快照里的 Teleport 代码集中在 `src/utils/teleport.tsx` 与 `src/utils/teleport/`。它会验证环境、获取或创建远程会话、请求远端日志、协调 Git bundle 与分支信息、再把恢复出的消息重新送回本地会话体系。这个过程本质上是在“对话恢复”和“代码工作区恢复”之间建立桥梁。

## 关键源码锚点

- `src/utils/teleport/api.ts`
- `src/utils/teleport/gitBundle.ts`

## 快照修正与补充

- `other-ans/ch26.md` 把 Teleport 定位为远程协作机制，这与当前快照中的 API、日志和 Git 处理路径一致。
- `../04-bridge-system.md` 讲的是 Bridge 与 Remote Control，不应与 Teleport 完全混为一谈；二者都涉及远程，但目标不同。
- Teleport 依赖 OAuth、远程 API 和 feature/gate 条件，因此在外部快照里更适合按接口与流程理解，而不是假设可以完整跑通。

## 设计启示

- 远程协作最难的是状态连续性，而不是“远端能启动一个进程”。
- 会话日志与 Git 状态必须一起恢复，才能让远程继续工作有意义。
- Teleport 体现的是“把编码会话当作可迁移对象”这一产品抽象。
