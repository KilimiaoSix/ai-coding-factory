# 第 28 章：全局状态设计

## 问题定义

终端 Agent 看起来像单线程对话，但实际上同时存在消息流、工具状态、权限上下文、UI 模式、任务状态和远程协作状态，因此需要一个明确的全局状态模型。

## 架构分析

当前快照采用轻量自研 store：`createStore()` 提供 `getState`、`setState`、`subscribe`，`AppStateProvider` 再用 React context 和 `useSyncExternalStore` 把它接进组件树。这种方式避免了引入重型状态框架，同时保持单向数据流和类型约束。

- 这里的“自研 Store”其实非常克制，本质是最小可用订阅模型，而非 Redux 式事件总线。

## 设计启示

- 终端 UI 再复杂，也不一定需要引入完整状态框架；关键是把可变状态收口到少数入口。
- store 轻量并不等于随意，反而要求状态切分更清晰。
- Agent 的全局状态应优先服务消息、权限和任务这三类核心流。
