# 第 42 章：可观测性——日志、遥测与诊断

## 问题定义

Agent 系统如果没有可观测性，就几乎无法定位“为什么这次 turn 很慢、为什么某个功能没有启用、为什么远程配置失效、为什么会话恢复异常”。Claude Code 因此把 analytics、GrowthBook、startup profiler、session log 和 diagnostics 都纳入基础设施层。

## 架构分析

`src/services/analytics/index.ts` 提供无依赖公共入口，把事件排队到 sink；GrowthBook 负责 feature/gate 与动态配置；1P/Datadog 等下游由 sink 和 exporter 承接。除此之外，还有 startup profiler、query profiler、session transcript、diagnostic logs 等更偏本地排障的通道。

## 关键源码锚点

- `src/services/analytics/index.ts`
- `src/services/analytics/growthbook.ts`
- `src/services/analytics/sink.ts`
- `src/utils/startupProfiler.ts`
- `src/utils/queryProfiler.ts`
- `src/utils/sessionStorage.ts`

## 快照修正与补充

- `other-ans/ch42.md` 把日志、遥测与诊断并列，这是准确的，因为当前快照里这些能力来自不同层：事件汇总、远程配置、性能采样和 transcript 持久化分别有独立入口。
- `../02-service-layer.md` 只把 analytics 视作服务，本章强调它与 feature gate、性能诊断和审计的关系。
- Analytics 元数据显式限制敏感字符串类型，这体现了可观测性与隐私保护同时被考虑。

## 设计启示

- 可观测性不仅用于产品指标，也用于控制复杂 Agent 系统的工程风险。
- 事件入口保持无依赖，是避免循环依赖和启动成本膨胀的好方法。
- 把 transcript 也看作诊断资产，会让问题定位能力提升一个层级。
