# 第 6 章：流式响应架构

## 问题定义

Claude Code 的用户体验建立在“边生成边显示”之上。模型响应不是一个完整字符串，而是一连串流式事件；UI、工具执行器、持久化和错误恢复都要与这条流协作。

## 架构分析

流式链路从 API SSE 开始，经 `callModel()` 逐块转成内部消息，再被 `query.ts` 消费。文本 token 会尽快送给 UI，工具块会触发 `StreamingToolExecutor` 或批量执行路径，可恢复错误会先被扣留，等待恢复逻辑决定是否继续。

## 关键源码锚点

- `src/services/api/claude.ts`
- `src/query.ts`
- `src/services/tools/`
- `src/remote/sdkMessageAdapter.ts`
- `src/utils/messages/mappers.ts`

## 快照修正与补充

- `other-ans/ch06.md` 对 withhold/backpressure 的解释是核心价值点；当前快照中这些行为确实贯穿 API 调用、工具执行和会话持久化。
- `../06-query-engine.md` 更偏主循环，本章补足的是“事件如何边到达边处理”。
- 流式架构不只服务 UI，也服务工具并发和错误恢复。

## 设计启示

- 流式响应应被视作系统级约束，而不是单个接口选项。
- 一旦模型输出是流式的，工具调度、存储和渲染就必须统一接受“部分完成”的现实。
- 可恢复错误的扣留机制，是流式系统优雅降级的代表性设计。
