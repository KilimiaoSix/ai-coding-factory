# 第 39 章：错误处理与重试策略

## 问题定义

Agent 系统面对的错误不仅有普通异常，还包括模型过载、网络闪断、OAuth 过期、上下文过长、输出 token 超限和工具执行失败。Claude Code 因此需要一套分层的错误恢复策略。

## 架构分析

API 侧的重试与回退主要由 `withRetry.ts` 管理，它区分 529、429、连接重置、OAuth 401、fast mode 冷却等不同场景。QueryEngine 与 query 循环则处理 prompt-too-long、max-output-tokens、工具错误隔离和流式扣留逻辑。换言之，重试不是单个函数行为，而是“服务层 + 循环层”的协作。

## 快照修正与补充

- `other-ans/ch39.md` 对错误恢复、模型回退、重试边界的总结，可以在当前 `withRetry.ts` 与 `query.ts` 中找到直接证据。
- `../02-service-layer.md` 只给出 API 错误处理概览，本章补足“错误类型如何影响继续、回退或终止”的运行时差异。
- 并不是所有错误都应自动重试，很多辅助查询源默认故障即放弃，以减少系统放大效应。

## 设计启示

- 错误处理要按来源和可恢复性分层，而不是统一 catch 后重试。
- 对 Agent 而言，“不给用户看到可恢复错误”本身是产品体验设计的一部分。
- 回退与重试策略必须考虑成本和放大效应，尤其是在服务过载时。
