# 第 9 章：QueryEngine——查询编排器

## 问题定义

`query.ts` 负责循环本身，但真正与 REPL、会话状态、系统提示、持久化和 usage 汇总打交道的是 `QueryEngine`。它相当于“单次查询的门面编排器”。

## 架构分析

`QueryEngine.submitMessage()` 的职责包括：处理用户输入、加载记忆与系统提示、写入 transcript、调用 `query()`、消费流式事件、记录 usage、更新内部消息缓存，以及在需要时处理中断和重放。换言之，它负责把“UI 里的输入”变成“可执行的 Agent turn”。

## 关键源码锚点

- `src/QueryEngine.ts`
- `src/query.ts`
- `src/utils/processUserInput/processUserInput.ts`
- `src/utils/queryContext.ts`
- `src/utils/sessionStorage.ts`
- `src/services/api/claude.ts`

## 快照修正与补充

- `../06-query-engine.md` 对 `QueryEngineConfig`、usage 和消息缓存有直接概览，本章将其定位为门面层而不是主循环本身。
- `other-ans/ch09.md` 中“生命周期、职责划分、消息规范化、预算检查”的结构与当前快照基本吻合。
- `QueryEngine` 的重要性在于整合，而不是在于拥有最多业务逻辑。

## 设计启示

- 把“编排器”与“循环执行器”拆开，可以降低主循环的耦合度。
- 门面层适合承接持久化、上下文构建和外部状态同步这类横切逻辑。
- 对 Agent 来说，`submitMessage()` 的边界比类名更重要，因为它决定了一次用户交互从哪开始、到哪结束。
