# 第 40 章：成本控制与速率限制

## 问题定义

Claude Code 不只是“能否完成任务”的系统，它还需要告诉用户和产品自己：这次会话花了多少 token、多少钱、多久，是否触发配额或速率限制，以及是否该进入 fast mode 或 extra usage 路径。

## 架构分析

成本与配额相关逻辑分布在 `cost-tracker.ts`、usage API、Claude AI limits、fast mode 和 rate limit 相关服务中。会话级成本会被写回项目配置，REPL 则通过状态与消息组件向用户呈现当前成本、限额和警告信息。

## 关键源码锚点

- `src/cost-tracker.ts`
- `src/services/api/usage.ts`
- `src/services/claudeAiLimits.ts`
- `src/utils/fastMode.ts`
- `src/components/CostThresholdDialog.tsx`
- `src/commands/usage/`

## 快照修正与补充

- `other-ans/ch40.md` 将成本控制和限流视为韧性的一部分，这在当前快照中成立，因为 usage、cost 和 rate limit 都会反向影响查询策略与 UI 反馈。
- `../02-service-layer.md` 没有完整展开这一块，本章补足了产品层成本反馈与后端 usage API 之间的连接。
- 外部快照中某些配额特性取决于 OAuth 订阅状态和远程接口可用性，不是纯本地逻辑。

## 设计启示

- 成本追踪最好成为会话内建能力，而不是事后统计。
- 配额与速率限制不仅影响后端调用，也应影响前端提示和用户决策。
- 对编程 Agent 来说，长会话成本透明度会直接影响用户信任。
