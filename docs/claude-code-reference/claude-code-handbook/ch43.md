# 第 43 章：启动流程——从命令行到 Agent 就绪

## 问题定义

Claude Code 的启动并不是“解析参数然后跑主程序”这么简单。它需要在尽量短的启动延迟内完成 fast path、配置启用、安全环境变量、网络预热、遥测初始化和 REPL 准备。

## 架构分析

启动链至少分成三段：`entrypoints/cli.tsx` 负责最轻的 fast path 和特定模式分流；`main.tsx` 负责完整 CLI 初始化与模块拼装；`entrypoints/init.ts` 负责配置、远程设置、代理、mTLS、预连接、OAuth 资料、scratchpad 和 telemetry 等运行时基础设施初始化。

```mermaid
flowchart LR
  A["cli.tsx"] --> B["fast path / 特殊入口"]
  B --> C["main.tsx"]
  C --> D["init.ts"]
  D --> E["REPL / QueryEngine / Tools 就绪"]
```

## 关键源码锚点

- `src/entrypoints/cli.tsx`
- `src/main.tsx`
- `src/entrypoints/init.ts`
- `src/bootstrap/state.ts`
- `src/commands.ts`
- `scripts/build.ts`

## 快照修正与补充

- `../01-entrypoint-and-startup.md` 已详细说明 `--version` 等 fast path，本章在此基础上补上 `init.ts` 负责的网络、设置和 telemetry 预热。
- `other-ans/ch43.md` 把启动看成“从命令行到 Agent 就绪”的生命周期，这与当前三段式启动链完全一致。
- 很多启动分支是按 feature gate 或产品模式拆开的，外部快照里并不会全部实际激活。

## 设计启示

- 终端 Agent 的启动优化值得单独设计，因为用户对首屏延迟极其敏感。
- 把 fast path 留在极薄入口层，是减少冷启动成本的关键。
- 初始化应尽可能并行且可降级，否则复杂能力会拖垮基础交互。
