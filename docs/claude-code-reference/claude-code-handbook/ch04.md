# 第 4 章：技术选型与权衡

## 问题定义

Claude Code 的技术栈表面上是 TypeScript、React、Ink、Bun、Commander.js，但真正值得分析的是：这些选型如何塑造了今天这份代码快照的模块边界与约束。

## 架构分析

- TypeScript：让 Tool、Command、Message、Permission 这类跨层接口能保持强类型约束。
- React + Ink：把终端 UI 从命令式输出改造成声明式组件树，适合处理流式消息、输入框、弹窗和任务状态。
- Bun：既提供 bundler，也通过 `bun:bundle` 让 feature flag 成为构建时的死代码边界。
- Commander.js：负责 CLI 参数与子命令解析，降低入口层复杂度。
- 自研/轻量状态层：当前快照没有引入重型状态框架，而是把状态、signal、订阅器散布在 `src/state/` 与 `src/utils/` 中。

## 关键源码锚点

- `scripts/build.ts`：最能体现 Bun 和 feature gate 的构建价值。
- `src/main.tsx`：Commander、React、Bun 门控在同一入口汇合。
- `src/ink/reconciler.ts`：React 19 + 自定义宿主环境。
- `src/Tool.ts`：TypeScript 类型约束的核心收益所在。
- `src/state/`

## 快照修正与补充

- `other-ans/ch04.md` 把“为什么不是 Go + BubbleTea”作为对照，这个论证方向是成立的，但当前快照里真正能直接验证的是 TypeScript 与 React 带来的接口统一性。
- `../01-entrypoint-and-startup.md` 也显示，Bun 不是单纯的运行时替代，而是 feature gate、宏注入、内部包桩实现的一部分。
- 当前仓库的“外部快照可构建”很大程度上依赖 `scripts/build.ts` 对内部包和缺失源码的兜底策略。

## 设计启示

- 选型不是“为了先进”，而是为了让 Agent 的高变动控制流可维护。
- Bun 在这里最重要的价值不是快，而是把“不同发行版能力”压进一个统一构建边界。
- 终端 Agent 只要 UI 足够复杂，React 的价值就会迅速超过传统 CLI 渲染方式。
