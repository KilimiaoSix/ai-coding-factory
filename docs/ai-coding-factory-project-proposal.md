# AI-Coding-Factory 项目立项文档

## 一、项目概述

### 1.1 项目名称

AI-Coding-Factory

### 1.2 项目定位

AI-Coding-Factory 是一个面向软件研发组织的多 Agent 自动化研发平台。项目目标不是简单做一个“更强的 AI 编码助手”，而是构建一套具备任务拆解、技术评审、并行开发、自动验证、缺陷回流、质量门禁与可视化管理能力的 AI 研发流水线。

该平台将用户需求、产品设计、技术方案、代码实现、测试验证、Bug 修复与交付上线组织为一套可持续运行的生产系统，使 AI 在研发过程中扮演类似“产品经理、技术负责人、开发工程师、测试工程师”的角色，并允许用户对这些 Agent 进行编辑、约束和治理。

### 1.3 项目愿景

将“人与 AI 对话式写代码”升级为“人管理一支可控的 AI 研发团队”，逐步形成面向需求交付的 AI 编程公司基础设施。

## 二、立项背景

### 2.1 行业背景

当前 AI 编码工具已经能够显著提升单点编码效率，但大多数工具仍停留在“单 Agent + 单轮对话 + 局部代码修改”的阶段。对于真实研发流程中常见的需求分析、任务拆解、技术可行性评估、跨模块协作、测试回归、缺陷回流和质量门禁，现有工具普遍缺少系统性的组织能力。

### 2.2 当前痛点

研发团队在使用 AI 过程中，通常面临以下问题：

- AI 擅长写代码，但不擅长长期保持跨任务上下文与统一目标
- 需求、方案、实现、测试之间缺少统一状态管理，容易出现“代码完成但未验证”或“任务关闭但证据不足”
- 多 Agent 协作往往停留在实验性质，缺少稳定的任务分发、回收与问责机制
- 质量控制仍高度依赖人工盯防，无法形成规模化复制的自动研发闭环
- AI 产生的大量过程知识未沉淀为组织资产，后续难以复用

### 2.3 项目机会

本项目具备良好的起点：

- 现有代码库已经具备子 Agent、任务、消息回传、Agent 编辑等底层能力
- `Dark Factory` 文档已经总结出适合 AI 长时间稳定运行的开发模式
- `auto-coding-agent-demo` 与 `hello-nextjs` 已验证“任务外置 + 强约束流程 + 自动测试 + 过程留痕”的最小闭环是可行的
- `BMAD Method` 提供了可参考的阶段化流程、角色分工、项目上下文和评审方法标准

因此，当前最有价值的方向不是继续增强单个 Agent 的能力，而是基于已有内核向上构建“多角色、多阶段、可审计、可视化”的 AI 研发工厂。

## 三、项目目标

### 3.1 总体目标

构建一个面向需求交付的 AI 研发平台，使用户可以：

- 提交需求文档，或以交互方式持续补充需求
- 先由 Product Agent 审阅需求文档或通过对话澄清需求，整理目标、范围与验收标准
- 在用户确认无误并点击 `Start` 后，由系统冻结一版需求基线、创建正式 `Requirement`，并进入工厂内部正式评审流程
- 由 PM Agent 负责项目推进、依赖协调、风险跟踪与流程升级
- 由技术 Lead Agent 自动进行方案设计、技术可行性验证与必要调研
- 将可执行任务分发给开发 Agent 并行开发
- 将开发结果自动交给测试 Agent 验证，发现问题后形成 Bug 回流
- 在通过质量门禁后形成交付物并反馈给用户
- 创建、编辑、约束不同角色 Agent，形成企业级 AI 研发组织能力

### 3.2 首期建设目标

项目首期不追求“一键自动上线”，重点建设以下能力：

- 需求到任务的结构化拆解能力
- 多角色 Agent 编排能力
- 任务状态机与证据留痕能力
- 自动验证与 Bug 回流能力
- 强 GUI 控制台与 Agent 编辑能力

首期范围聚焦于 Web 项目与后端服务项目的研发闭环，不纳入移动端、游戏、真机/模拟器农场等高复杂度测试场景。

### 3.3 首期测试环境范围

为保证验证结果具备工程可信度，测试环境不应隐含在某个 Agent 的 Prompt 或脚本中，而应作为平台级可配置能力独立建设。

项目首期仅建设以下两类测试环境：

- `Backend Test Environment`：面向 API、数据库、缓存、消息队列、任务服务等后端能力验证
- `Web Test Environment`：面向浏览器页面渲染、用户交互流程、端到端验证

项目首期暂不纳入以下测试环境：

- Android 模拟器与 iOS 模拟器
- 游戏运行环境与游戏自动化测试
- 云真机、桌面客户端、复杂硬件依赖环境

这一范围控制有利于先把“需求 -> 开发 -> 后端验证 -> Web 验证 -> Bug 回流 -> 交付”这一核心闭环做稳定。

### 3.4 外部方法标准利用目标

项目将吸收外部成熟方法标准作为流程设计输入，其中首要参考标准之一为 `BMAD Method`。

对 BMAD 的利用目标不是直接照搬其全部工作流与文件体系，而是提炼其中适合 AI-Coding-Factory 的部分，用于增强项目的方法完整性与实施规范性。重点吸收方向包括：

- 分阶段工作流思想
- 角色分工模型
- 项目上下文初始化机制
- Implementation Readiness 检查思想
- 对抗性评审与测试治理思路

相关详细评估见：[bmad-standard-evaluation.md](/Users/kilimiao/CodeProject/claude-code/docs/bmad-standard-evaluation.md)

## 四、项目实施价值

### 4.1 对研发效率的价值

AI-Coding-Factory 将显著减少研发流程中的串行等待与上下文切换成本：

- 产品分析、技术调研、开发实现、验证测试可以部分并行展开
- 常规需求能够从“人工逐段驱动”转变为“流程驱动 + 例外介入”
- 对重复性较高的任务，例如 CRUD、页面搭建、接口封装、测试补齐、文档更新等，平台可持续复用成熟工作流

其核心价值不只是“写代码更快”，而是“需求交付链路更短、节奏更稳定、吞吐更高”。

### 4.2 对质量控制的价值

项目将质量控制前置并系统化，而不是依赖开发者在末端补救：

- 每个任务都要求明确验收标准
- 每个阶段都要求可审计的交付证据
- 测试 Agent 与开发 Agent 角色分离，降低“自证正确”的风险
- Bug 回流成为正式流程，而不是临时人工提醒
- 通过状态机控制“未验证不得完成、未达标不得交付”

这将把当前依赖经验与自觉的质量约束，升级为平台级的强约束机制。

同时，将测试环境独立配置还能避免“开发环境即测试环境”的混乱局面，使后端与 Web 验证具备更稳定、更接近真实运行条件的执行基础。

### 4.3 对组织能力的价值

项目的长期价值在于把 AI 使用方式从“个人技巧”升级为“组织能力”：

- 将高质量 Prompt、角色规范、工作流模板沉淀为团队资产
- 让不同项目能够复用同一套研发角色与治理机制
- 降低对个别高级使用者的依赖，提升团队整体 AI 生产效率
- 建立标准化的 AI 研发制度，为后续规模扩张提供基础

### 4.4 对知识沉淀的价值

传统对话式 AI 容易产生大量不可复用的临时上下文，而本项目会把关键知识转为结构化资产：

- 需求澄清记录
- 技术评审结论
- 任务执行状态
- 测试与缺陷证据
- Agent 配置与角色规范

这些内容一旦沉淀下来，将形成研发组织的“机器可执行知识库”。

## 五、项目实施可行性分析

### 5.1 技术可行性

本项目具备较高技术可行性，原因如下：

#### 1. 已有执行内核可复用

当前代码库已经具备多项关键底层能力：

- 子 Agent 启动与后台任务执行
- 任务创建、更新、状态流转与所有权分配
- Agent 间消息传递与结果回收
- Agent 定义、创建与编辑
- 结构化会话事件输出

这意味着项目无需从零构建 Agent Runtime，而是可以在现有能力之上新增控制平面、状态机和 GUI。

对于首期范围内的测试执行，技术路径也较为明确：

- 后端侧可以采用 `Docker Compose` 或等价容器编排方式构建接近真实依赖的隔离验证环境
- Web 端侧可以采用浏览器自动化工具构建页面级与流程级验证能力
- 两类环境都可以通过统一的“测试环境配置”接入平台，而不要求底层工具完全一致

#### 2. 已有流程模式可复用

`Dark Factory` 模式已经证明，AI 开发的稳定性不依赖单次 Prompt 强度，而依赖：

- 任务状态外置
- 单任务闭环
- 强制验证
- 显式阻塞
- 原子提交

这为本项目提供了成熟的方法论基础。

同时，BMAD Method 提供了更完整的角色体系与阶段化工作流参考，可作为项目上层方法标准的补充来源，用于规范产品分析、架构评审、任务拆解与实施准备度检查等环节。

#### 3. 已有 Demo 可验证

`auto-coding-agent-demo` 和 `hello-nextjs` 已经证明：

- 通过任务文件驱动开发是可行的
- AI 可以完成一个包含前端、后端、数据库、第三方 API 集成的完整项目
- 浏览器测试、构建验证、阻塞回滚、人工介入等真实工程情境可以被纳入流程

这说明项目并非概念验证阶段，而是已经具备向平台化演进的现实基础。

### 5.2 工程可行性

从工程建设角度看，本项目适合分阶段推进：

#### 第一阶段：控制平面 MVP

- 需求录入
- 任务拆解
- Agent 角色定义
- 任务状态可视化
- 开发与测试回流
- Web 与后端测试环境配置接入

#### 第二阶段：质量门禁与证据系统

- 验证报告结构化
- Bug 生命周期管理
- 阶段准入条件
- 交付就绪判断

#### 第三阶段：组织化运营能力

- Agent 市场与模板
- 项目级知识沉淀
- 角色权限治理
- 统计看板与效率分析

该分期方式可以避免大而全一次性建设失败，先验证核心闭环，再逐步增强组织能力。

在工程实现上，建议将测试执行基座建模为独立模块，而不是散落在各类 Agent Prompt 中。平台只要求 Agent 声明所需验证能力，由测试环境配置决定底层如何启动依赖、运行脚本、收集产物和返回证据。

#### 控制平面真相源与本地优先架构决策

考虑到 AI-Coding-Factory 首期将以客户端形态为主，且核心执行面天然位于本地工作区、测试环境和代码仓库之中，项目控制平面应采用“本地优先”的实现策略，而不是从一开始就依赖服务端作为唯一真相源。

首期明确采用以下控制平面决策：

- 结构化状态以本地数据库为主保存，优先采用 `SQLite` 等适合桌面应用的嵌入式数据库
- 日志、截图、测试报告、构建产物等大对象证据以本地文件系统保存
- 数据库只保存结构化控制状态与证据索引，不承担大文件本体存储
- GUI、Agent、测试环境和门禁逻辑统一以本地数据库中的结构化状态作为主要读取依据
- 未来如引入团队协作能力，服务端主要承担同步、共享视图、审计与组织治理职责，而不是替代本地控制平面成为首期唯一状态源

这一决策有助于同时满足以下目标：

- 保证客户端离线可用与本地执行闭环
- 让项目状态与本地代码仓库、测试环境和产物天然对齐
- 避免首期因服务端依赖、同步一致性和路径映射问题而显著放大工程复杂度
- 为后续团队协作预留清晰扩展边界，使服务端能力建立在本地优先架构之上，而不是推翻首期实现

在具体职责划分上，首期建议采用“本地数据库 + 本地 Artifact Store”的混合模式：

- 本地数据库负责保存需求、任务、验证结果、Bug、阻塞状态、阶段门禁、Agent 配置、测试环境配置等结构化控制平面数据
- 本地文件系统负责保存日志、截图、视频、构建产物、测试报告和其他大体量证据
- 两者通过稳定 ID 和索引关系关联，为后续同步与协作能力提供基础

#### 开发任务隔离与 worktree 策略决策

考虑到 AI-Coding-Factory 的核心目标之一是支持多 Agent 并行开发，首期必须从一开始就解决写任务互相污染、验证结果无法归因、回滚困难等工程问题。因此，项目首期明确采用“写任务默认 worktree 隔离”的策略。

首期具体规则如下：

- `Dev Agent` 承担的写代码任务默认分配独立 `worktree`
- 纯阅读、分析、方案设计、需求整理等非写任务不强制进入 `worktree`
- `QA Agent` 若验证的是某个具体开发任务的产出，应优先在对应任务的 `worktree` 上执行验证
- 是否允许人工或策略层 override 可作为扩展能力保留，但不作为首期默认执行路径

这一策略的核心目的包括：

- 让并行开发任务具备清晰的代码隔离边界
- 让验证结果、产物证据与具体任务保持稳定归因关系
- 降低不同 Agent 互相覆盖文件和污染运行环境的风险
- 提高任务回滚、回收与后续集成的可控性

#### QA、Bug 体系与发布裁决边界决策

首期项目需要尽早明确质量治理中的角色边界，避免系统退化为“开发自证正确”或“测试只负责跑命令、不进入流程控制”的弱治理模式。

首期明确采用以下规则：

- `QA Agent` 不只是验证执行者，同时承担质量负向裁决职责
- `QA Agent` 是 `Bug` 体系的唯一创建者、回归验证者与关闭者
- `Dev Agent` 在开发过程中发现的问题不纳入 `Bug` 体系，而作为当前任务内部返工处理
- `Dev Agent` 负责修复由 `QA Agent` 提出的 `Bug`，但无权直接关闭 `Bug`
- `QA Agent` 有权基于验证失败创建 `Bug`、触发回归验证，并让相关 `Gate` 进入 `failed` 或 `blocked`
- `QA Agent` 无权推进阶段，也无权做最终发布决定
- 阶段推进不应由单一角色独占，而应由系统聚合事实后交由对应阶段的主控 Agent 做 `GateDecision`
- `Lead Agent` 负责技术后段的主控裁决与发布建议，但不再独占所有阶段推进权

为保证发布体系稳定，首期还应同时遵循以下约束：

- 只要存在 `open` 或 `pending_qa_retest` 的阻断型 `Bug`，相关 `Gate` 不得通过
- 任一主控 Agent 的阶段推进或发布相关决策都必须建立在结构化控制平面状态之上，而不能仅依据单个 Agent 的口头结论
- `QA Agent` 具备否决输入权，但不具备发布批准权
- `Dev Agent` 具备实现与修复权，但不具备缺陷关闭权和发布裁决权

#### Backend Test Environment 与 Web Test Environment 的实现分层决策

首期项目不应为了追求形式上的整齐而过早强行统一 `Backend Test Environment` 与 `Web Test Environment` 的底层执行器实现。两类环境虽然同属“测试环境”，但在依赖拉起、运行机制、证据采集和失败模式上存在显著差异。

因此，首期明确采用“底层分别实现、上层统一结果模型”的策略：

- `Backend Test Environment` 与 `Web Test Environment` 在执行器层分别实现
- 控制平面统一消费 `VerificationRun`、`Evidence`、`FailureCategory`、`GateDecision` 等结构化结果
- 首期不强求两类环境在启动命令、依赖编排、验证脚本和证据采集方式上保持一致
- 首期优先统一的是状态语义、结果结构和门禁消费接口，而不是底层运行细节

这一策略可以带来以下收益：

- 降低首期因过早抽象导致的实现复杂度
- 让后端验证与 Web 验证都能按各自最适合的方式稳定落地
- 保证 Bug 回流、门禁判断和 GUI 展示使用统一治理语言
- 为后续在共性稳定后再提炼共享抽象预留空间

#### GUI 产品形态与首期交互边界决策

首期 GUI 应被定义为“多项目老板视角的运营控制台”，而不是单纯的对话窗口，也不是追求视觉复杂度的工厂大屏。其目标是让用户能够从组织管理视角观察项目、Agent、任务、缺陷、验证和发布状态，并基于这些状态进行需求委派、治理与裁决。

首期 GUI 建议满足以下能力：

- 支持查看多个开发项目，并在项目级维度隔离工作区、任务状态、缺陷状态和环境配置
- 支持查看各角色 Agent 的当前工作状态、所属任务、运行摘要、阻塞情况与模型配置
- 支持用户通过与 `Product Agent` 对话或提交需求文档逐步形成需求，并在确认后点击 `Start` 正式启动工厂流程
- 支持为不同角色 Agent 配置不同模型，并允许项目级覆盖
- 支持设置界面，用于配置测试环境及相关执行前提

在架构上，首期应遵循以下约束：

- `Start` 前需求处于澄清态，且仅由 `Product Agent` 维护 `RequirementDraft`；点击 `Start` 后由系统冻结需求基线并生成正式 `Requirement`
- `Start` 不表示直接进入开发执行，而是表示正式进入工厂流程；新建的 `Requirement` 从 `phase = intake` 开始，由 `Product Agent`、`PM Agent` 与 `Lead Agent` 进入正式需求评审
- 用户与 `Product Agent` 的对话必须落到同一套结构化控制平面对象上，而不是停留在不可追踪的会话上下文中
- GUI 顶层优先展示 Requirement、Phase、Gate、Task、Bug、Verification 等业务状态，其次才是底层执行细节
- 首期不优先建设复杂流程编辑器、图形化编排画布或重视觉化大屏
- 首期 GUI 的核心价值在于“可观察、可委派、可裁决、可配置”，而不是视觉炫技

### 5.3 业务可行性

本项目具备明确的落地方向：

- 适用于需求明确、可拆分、可验证的软件项目
- 适合作为内部研发提效平台率先试点
- 适合在中小项目、原型开发、内部工具、标准业务模块等场景快速产生价值
- 后续可扩展到更复杂的研发项目与企业级流程治理

通过引入 BMAD 这类外部方法标准作为流程模板来源，项目还能降低“完全从零定义流程标准”的成本，加快首期试点方法收敛速度。

### 5.4 组织可行性

项目不会要求组织立即改变所有研发方式，而可以以“AI 辅助工厂”模式逐步落地：

- 初期由人主导需求和关键决策，AI 负责执行性工作
- 中期由 AI 负责更多拆解、验证和回流，人负责治理与例外处理
- 后期逐步沉淀为标准化团队能力

这种渐进式落地方式，组织阻力低，试点成功率更高。

## 六、项目可带来的收益

### 6.1 直接收益

#### 1. 缩短需求交付周期

在流程设计合理的前提下，项目有望带来以下直接收益：

- 减少需求理解到任务执行之间的人工搬运成本
- 将技术调研、任务拆解、开发与验证部分并行化
- 减少重复性编码与重复性测试的人力占用

对于标准化程度较高的项目，需求交付周期有望显著缩短。

#### 2. 提升研发吞吐

传统研发的瓶颈往往来自关键人并发能力有限。AI-Coding-Factory 的核心收益之一，是把单个研发人员的“并发上限”提升为“可管理的多 Agent 并发能力”。

即便人类仍然是最终责任人，也可以把更多精力放在目标设定、边界管理和关键判断上，而不是反复亲自执行低杠杆工作。

#### 3. 降低返工成本

通过显式验收、测试回流和阶段门禁，项目可以更早暴露问题，降低后期集中返工的概率。相较“先写完再补验证”的方式，这种流程对复杂项目更稳健。

### 6.2 间接收益

#### 1. 构建企业级 AI 研发资产

长期来看，项目沉淀的最大价值不是代码本身，而是：

- 角色模板
- 工作流模板
- 项目结构模板
- 验证规则
- 历史决策与知识资产

这些将成为组织可复用、可复制、可扩展的能力底盘。

#### 2. 形成研发治理能力

项目能够让 AI 研发行为从“黑盒式输出”变成“流程可见、状态可见、证据可见、责任可见”的治理对象。对于后续的质量审计、项目管理、风险控制都具有长期价值。

#### 3. 提升组织对 AI 的掌控力

项目不是让团队依赖某个“会写 Prompt 的高手”，而是把高质量 AI 使用方式产品化、制度化、平台化。这会显著提升组织层面对 AI 的掌控程度。

### 6.3 可量化收益建议

项目试点期建议重点跟踪以下指标：

- 需求从录入到首版可运行交付的平均时长
- 单需求的人类投入时长
- 单需求的 Agent 执行轮次
- 首轮验证通过率
- Bug 回流次数与修复周期
- 可复用 Agent 模板数量
- 需求交付成功率

这些指标能够帮助项目从“概念上有价值”走向“结果上可衡量”。

## 七、实施风险与应对策略

### 7.1 主要风险

#### 1. 需求不清导致错误放大

AI 会高效执行错误目标，因此需求入口必须具备澄清和约束机制。

#### 2. 角色边界不清导致协作失控

如果产品、技术、开发、测试 Agent 的职责划分不明确，系统会退化回“多个会写代码的 Agent 相互干扰”。

#### 3. 验证不足导致假完成

如果没有结构化验证标准与门禁，平台会高频出现“看起来完成，实际上不可交付”的问题。

#### 4. 外部依赖阻塞

账号、密钥、部署环境、第三方服务等问题仍然需要人工介入，必须设计标准化阻塞处理流程。

#### 5. 测试环境复杂度失控

如果在首期同时覆盖后端、Web、移动端、游戏、真机和模拟器，项目复杂度会迅速膨胀，导致验证链路难以稳定。

#### 6. 外部标准照搬导致系统失配

如果直接照搬外部方法体系的全部工件形态、命令结构和运行方式，可能导致平台架构被外部标准反向绑定，削弱系统自身的一致性与演进能力。

### 7.2 应对策略

- 引入需求澄清和验收标准生成功能
- 建立严格的角色定义与输出模板
- 将验证结果结构化并作为阶段门禁条件
- 将阻塞状态、所需人工动作和解除条件明确记录
- 将测试环境作为平台级配置项管理，而不是在 Agent 中硬编码
- 首期仅覆盖 Web 与后端测试环境，暂缓高复杂度端侧
- 对 BMAD 等外部标准采用“语义吸收、形态重构”的方式，保留方法价值，不照搬运行形态
- 先从可验证、可复用、高价值的场景试点，逐步扩大范围

## 八、项目建设建议

### 8.1 建设原则

- 以现有执行内核为基础，优先新增控制平面，而非重写底层运行时
- 以 Dark Factory 闭环为流程核心，优先保证任务、验证、留痕、回流可用
- 以 GUI 为组织控制台，而不是仅作为终端能力的视觉包装
- 以“可验证、可审计、可复用”为首要目标，而不是追求表面上的全自动
- 以“本地优先控制平面”为原则，结构化状态优先落本地数据库，证据优先落本地文件系统，未来服务端提供同步与协作扩展
- 以“写任务默认 worktree 隔离”为原则，优先保证多 Agent 并行开发时的代码隔离、验证归因与回滚能力
- 以“QA 负责缺陷认定与否决、阶段推进采用主控 Agent 机制、Lead 负责技术后段与发布建议”为原则，防止系统退化为开发自证正确或单角色独占流程推进
- 以“`GateDecision` 按阶段出口建模”为原则，按 phase transition 记录主控角色、必要输入、裁决结果与重判链路
- 以“评审链单独建模、回溯能力有限收敛”为原则，首期支持正式评审留痕与阶段重开，不做任意历史状态回滚
- 以“`Start` 后需求变更对象化”为原则，用户输入持续开放，但影响执行基线的变更必须通过独立 `Change Request` 受控进入流程
- 以“用户接入型 Blocker 升级权分角色、前台表达统一”为原则，技术型归 Lead、协调型归 PM、业务决策型归 Product，但统一由 Product Agent 面向用户表达
- 以“测试环境底层分治、上层统一结果模型”为原则，优先保证验证闭环稳定，而不是过早统一执行器实现
- 以“GUI 作为多项目运营控制台”为原则，优先强化可观察、可委派、可裁决、可配置能力
- 以“正式 `Requirement` 按 `Product / PM / Lead / System` 分层建模”为原则，避免业务定义、项目推进、技术约束与系统摘要混写
- 以“测试环境独立配置”为原则，避免把环境逻辑散落到 Agent Prompt 和临时脚本中
- 以 BMAD 等外部方法标准作为流程设计参考，而不是作为平台架构的直接替代品

### 8.2 首期建议范围

建议首期聚焦以下 MVP 能力：

- 用户提交需求文档或交互式提需求
- Product Agent 作为前台入口，审阅需求文档或通过对话澄清需求，并整理 `RequirementDraft`
- `Product Agent` 判断需求已具备开工基础后，用户在确认需求摘要后点击 `Start`
- `Start` 由系统冻结需求基线并生成正式 `Requirement`，随后进入 `intake` 阶段的正式需求评审
- 正式 `Requirement` 按 `Product / PM / Lead / System` 分层字段建模，支撑需求评审、项目推进、技术可行性判断与系统聚合摘要
- `GateDecision` 按 `intake_to_planning / planning_to_execution / execution_to_qa / qa_to_release / release_to_done` 五类阶段出口建模，并采用 JSON 记录必要输入与输入满足情况；各 gate 使用一组统一命名的标准输入项
- 首期支持正式评审记录与阶段重开能力，保证需求评审、方案评审、业务验收与阶段裁决可追溯
- `Start` 后用户仍可持续输入，但影响执行基线的需求变化必须进入独立 `Change Request`；未获批前不自动打断在途 Agent
- 用户接入型 `Blocker` 按类型分配升级权，但最终统一由 `Product Agent` 面向用户表达，避免前台入口分裂
- PM Agent 负责项目推进、依赖协调、风险跟踪与流程升级
- Tech Lead Agent 输出技术方案与实施拆分
- Dev Agent 按任务执行开发，写任务默认在独立 `worktree` 中运行
- QA Agent 独立验证、创建与关闭 `Bug`，并负责质量回流与门禁否决输入
- 各阶段按主控 Agent 机制推进：PM Agent 主控前段流程推进，Lead Agent 主控技术后段推进与发布建议，Product Agent 负责业务验收输入
- 测试环境配置中心，首期支持 `Backend Test Environment` 与 `Web Test Environment`
- `Backend Test Environment` 与 `Web Test Environment` 底层分别实现，上层统一验证结果模型与门禁消费接口
- 后端测试环境支持依赖服务拉起、构建、lint/typecheck、集成测试与 API smoke test
- Web 测试环境支持前端应用拉起、浏览器自动化、截图/日志采集与核心流程 E2E
- GUI 作为多项目运营控制台，支持项目切换、Agent 状态观察、任务流与阶段状态查看、缺陷回流查看，以及与 `Product Agent` 的持续需求对话和 `Start` 启动交互
- 支持为不同角色 Agent 配置不同模型
- Agent 编辑器用于调整不同角色的能力边界
- 设置界面用于配置测试环境与相关执行前提
- 基于 BMAD 启发构建默认角色模板与阶段模板，但统一纳入 AI-Coding-Factory 的结构化数据模型

首期明确不做：

- 移动端模拟器与真机测试编排
- 游戏运行环境与游戏自动化测试
- 云设备农场与复杂硬件依赖验证
- 面向所有客户端形态的一体化测试基座
- 直接照搬 BMAD 全量文件体系与全部原生命令式工作流

### 8.3 成功标准

若项目在首期试点中达到以下标准，即可视为立项成功：

- 至少有一类真实项目可稳定跑通完整闭环
- 需求交付效率较传统流程有明显提升
- 关键质量问题能够在上线前被流程拦截
- 角色模板和流程模板能够跨项目复用

## 九、立项结论

AI-Coding-Factory 具备明确的业务价值、现实的技术基础和渐进式落地路径，适合正式立项推进。

其核心意义不在于再造一个 AI 编码工具，而在于建设一套面向组织交付的 AI 研发生产系统。该系统一旦形成，将同时带来以下三类收益：

- 短期收益：提升研发效率、缩短交付周期、降低重复劳动
- 中期收益：建立质量门禁、减少返工、提高需求交付稳定性
- 长期收益：沉淀企业级 AI 研发资产，形成可复制、可治理、可扩张的 AI 编程组织能力

综合判断，AI-Coding-Factory 项目具备较强的实施必要性与较高的实施可行性，建议按“控制平面 MVP -> 质量门禁 -> 组织化能力”的路径分阶段启动。
