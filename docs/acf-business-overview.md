# ACF 项目业务说明

## 1. 项目定位

`ACF` 的正式名称是“基于 ClaudeCode 实现的全自动代码工厂”。它不是单一 Agent 的功能增强版，而是一套建立在现有 Claude Code 基座之上的研发控制平面，用来组织需求、角色、任务、验证、缺陷和交付流程。

## 2. 目标用户

### 2.1 团队管理者

- 需要知道需求当前处于哪个阶段
- 需要看到验证结果、缺陷状态和关键风险
- 需要通过正式门禁而不是聊天上下文来推进交付

### 2.2 一线使用者

- 需要通过统一前台入口表达需求
- 需要收到结构化的澄清、阻塞反馈和变更评估
- 需要在不直接操心底层多 Agent 编排的情况下驱动交付

## 3. 使用场景

`ACF` 的典型场景是：用户提出需求，系统将需求冻结为正式对象，拆解为任务并分配给不同角色 Agent 完成实现与验证，再通过缺陷回流和阶段 Gate 完成交付。

这条主链路不是“对话结束即完成”，而是：

`RequirementDraft -> Start -> Requirement -> Task/TaskRun -> VerificationRun -> Bug/Blocker -> GateDecision -> Delivery`

## 4. 核心闭环

### 4.1 需求入口

用户先与 `Product Agent` 交互，形成需求草稿、项目画像和 readiness 检查结果。`Start` 之后，需求进入正式控制平面。

### 4.2 执行编排

`PM` 与 `Lead` 负责规划、拆解和推进。`Dev` 负责实现与修复，`QA` 负责验证、缺陷确认和回归。

### 4.3 验证回流

首期只聚焦 `Backend` 和 `Web` 两类验证基座，所有验证都要沉淀到统一结果模型中，不能停留在日志或聊天描述里。

### 4.4 质量门禁

阶段推进依赖正式 `ReviewRecord` 和 `GateDecision`，不能把“任务都做完了”直接等价为“已经可以交付”。

## 5. 用户价值

- 让需求、执行、验证和交付从聊天式过程变成结构化过程
- 让管理者获得正式的阶段状态、缺陷状态和证据链
- 让使用者获得统一前台入口，不必直接操心底层 Agent 协作细节
- 让已有 Claude Code 基座能力被组织成可治理的研发流程

## 6. 首期边界

首期只做最小闭环，不追求一次性覆盖所有平台与治理能力：

- 聚焦本地优先控制平面
- 聚焦 Backend 与 Web 两类验证环境
- 聚焦 Product、PM、Lead、Dev、QA 五类角色
- 聚焦 GUI 运营控制台，而不是复杂流程画布

不在首期范围内的内容包括移动端、游戏、真机农场、完整组织治理和服务端协作平台。
