# ACF 设计理念

## 1. 基于 Claude Code 基座继续开发

`ACF` 的起点不是重写 Claude Code，而是把当前仓库视为执行内核基座。这个基座已经提供了 Agent runtime、工具调用、会话留痕、任务原语、worktree 隔离能力和终端 UI 壳，足以支撑上层控制平面的建设。

因此，设计原则不是“替换基座”，而是“在稳定接入面之上增加控制平面”。

## 2. 控制平面与执行内核分层

执行内核负责：

- Agent 调用
- 工具执行
- worktree 原语
- transcript 和事件流

控制平面负责：

- 需求对象和阶段状态
- 评审、缺陷、阻塞和 Gate
- 验证结果汇总
- GUI 运营视图

这两层必须分开，否则运行时留痕会被误写成正式业务对象，系统会缺少稳定的阶段推进依据。

## 3. Dark Factory 的吸收边界

`ACF` 吸收 Dark Factory 的核心思想，但不直接照搬其外部表现：

- 吸收“长流程自动执行”和“验证先于宣告完成”的原则
- 吸收“任务、证据、状态、回流”必须闭环的要求
- 不把 demo 工作法直接等同于平台能力

Dark Factory 在这里是方法来源，不是控制平面对象设计的替代品。

## 4. BMAD 的吸收边界

`ACF` 吸收 BMAD 的重点是：

- 项目上下文初始化
- 实施准备度检查
- 角色分工与阶段推进

但首期只采用轻量机制，不做成沉重的方法论引擎。`ProjectContextSummary` 和 `ReadinessChecklist` 是前置约束，不是完整知识管理系统。

## 5. 本地优先与代码锚点

首期 `ACF` 采用本地优先方案。控制平面状态必须锚定到代码版本或 worktree 快照，不能只依赖聊天上下文。任何 `TaskRun`、`VerificationRun`、`GateDecision` 如果不能指向对应代码版本，其结论都不可靠。

## 6. 测试基座先分治、后统一

首期验证环境只保留 `Backend` 与 `Web` 两类。它们的底层运行方式可以不同，但最终都要汇总到统一的验证结果模型中。这样既能控制首期复杂度，也能为后续扩展留下接口。

## 7. Product Agent 是统一前台入口

用户不直接面向 PM、Lead、Dev、QA 多个角色交互，而是通过 `Product Agent` 进入系统。这样可以保证需求表达、阻塞反馈和变更沟通在外部只有一个稳定入口，避免角色表达分裂。

## 8. 阶段推进依赖正式门禁

在 `ACF` 中，任务完成不自动等价于阶段通过。阶段推进必须依赖正式的评审输入、验证结果、缺陷状态和 `GateDecision`。这是区分“自动执行系统”和“自动代码工厂”的关键边界。

## 9. 文档与结构也要体现分层

项目文档本身也应反映分层思想：

- `acf-blueprint.md` 负责唯一实施蓝图
- `acf-business-overview.md` 负责业务说明
- `acf-design-principles.md` 负责设计理念
- `claude-code-reference/` 负责基座参考资料
- `auto-coding-agent-demo/` 负责大型示例

文档结构必须服务于实施，不再把立项、评审、预设计拆成多个平行入口。
