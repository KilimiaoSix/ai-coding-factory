# Dark Factory 开发设计模式

本文档基于《Claude Code 电子书 & AI 全自动开发系统实战【数字游牧人懒人包】》整理，重点抽取并封装 `Dark Factory` 开发设计模式，便于在 AI Agent 驱动的软件工程场景中直接复用。

## 一、什么是 Dark Factory

`Dark Factory` 是一种面向 AI Agent 的自动化研发模式。它的核心目标不是让人更快地写代码，而是让模型在更长时间内稳定、持续、可验证地完成开发任务。

这套模式假设：

- 代码实现的主力已经从人转移到模型
- 人类的核心价值不再是亲自编码，而是设计流程、设置边界、拆分任务、处理阻塞
- 真正的工程杠杆不在单次 prompt，而在一套可持续运行的任务系统

可以把它理解为：把“和 AI 聊天写代码”升级成“让 AI 在一条受控产线上持续交付”。

## 二、核心理念

Dark Factory 在原始材料中强调“三不原则”：

- `No human writes code`：人不直接写代码
- `No human reviews code`：人不逐行审代码
- `No human tests code`：人不手动承担主要测试职责

这三条并不是说人完全退出，而是重新定义人的职责：

- 人负责设计流程和系统边界
- 人负责提供清晰任务和验收标准
- 人负责在 AI 被外部依赖、权限、账号、环境阻塞时介入

因此，Dark Factory 不是“完全放任 AI”，而是“用高约束流程替代人工盯防”。

## 三、设计目标

这套模式主要解决以下问题：

- 多轮会话中上下文衰减，导致 AI 忘记项目状态
- 任务交接不透明，不同 agent 无法共享真实进展
- 只产出代码，不记录测试和决策，后续无法接手
- 任务未验证就被标记完成，造成假进度
- 长时间开发缺乏统一节奏，AI 输出质量波动明显

Dark Factory 的设计目标就是把这些问题收敛为一条固定闭环：

`领任务 -> 实现 -> 验证 -> 记录 -> 标记完成 -> 提交`

## 四、最小系统组成

该模式可以只依赖三个核心文件运行。

### 4.1 `task.json`

职责：任务源和状态源。

作用：

- 存放待开发任务
- 明确哪些任务尚未通过验证
- 让不同 agent 基于同一份任务状态协作

最小示例：

```json
{
  "tasks": [
    {
      "description": "实现用户登录功能",
      "passes": false
    }
  ]
}
```

关键约束：

- AI 每次只领取一个 `passes: false` 的任务
- 只有在任务验证完成后，才能改成 `true`
- 不删除任务，不改写任务原意

### 4.2 `progress.txt`

职责：项目工作日志。

作用：

- 记录某次任务到底做了什么
- 记录如何验证
- 记录遗留问题和后续注意事项

最小示例：

```md
## 2026-04-09 - Task: 实现用户登录功能
### What was done:
- 创建登录页面
- 接入认证服务

### Testing:
- 浏览器流程测试通过
- 构建成功

### Notes:
- 真实环境变量仍需人工配置
```

关键价值：

- 让新 agent 不必重新阅读大量代码来恢复上下文
- 让任务状态不仅有“完成/未完成”，还有过程证据

### 4.3 `CLAUDE.md` 或 Agent 操作手册

职责：强制执行统一工作流。

作用：

- 规定 AI 每轮必须做什么
- 规定哪些测试是硬性要求
- 规定何时允许提交、何时必须停下

本质上，它是 `Dark Factory` 的工厂操作规程。

## 五、标准工作闭环

Dark Factory 推荐把每个 agent 会话限制为一个标准闭环。

### Step 1. 初始化环境

先运行项目初始化脚本，确保依赖、开发服务和运行前提具备。

典型动作：

```bash
./init.sh
```

目的：

- 降低环境差异
- 避免 agent 在错误环境下工作

### Step 2. 领取一个任务

读取 `task.json`，只选择一个 `passes: false` 的任务。

选择原则：

- 优先选未完成任务
- 优先选依赖更少、阻塞更少的任务
- 优先选能独立交付的小任务

核心规则：

- 一次只做一个任务
- 一个会话只完成一个任务

### Step 3. 实现任务

按照任务描述编写或修改代码，遵循现有代码风格和架构约束。

这一阶段不追求“顺手多做一点”，而强调：

- 范围小
- 目标清晰
- 不扩散需求

### Step 4. 强制验证

这是 Dark Factory 最关键的一步。任务不是“代码写完”就结束，而是“验证通过”才结束。

建议规则：

- 大幅 UI 修改必须做浏览器级测试
- 小范围逻辑调整至少要过 `lint` 和 `build`
- 所有变更都要有明确测试证据

典型验证项：

- TypeScript 无错误
- `lint` 通过
- `build` 成功
- UI 相关功能能在浏览器中正确运行

文档原意非常明确：没有验证，就不能标记完成。

### Step 5. 更新进度日志

在 `progress.txt` 中记录：

- 做了什么
- 怎么测试的
- 当前还剩什么风险或依赖

这一步的目的是把会话内的隐性上下文转成显性文档。

### Step 6. 原子化提交

当且仅当验证通过后，才允许：

- 更新 `task.json`，把对应任务从 `false` 改为 `true`
- 把代码变更、`progress.txt`、`task.json` 放在同一个 commit 中提交

核心思想是“任务状态更新必须与代码提交原子一致”。  
否则就会出现“代码改了但任务没关”“任务关了但代码没证据”的状态漂移。

## 六、阻塞处理机制

Dark Factory 不是要求 AI 硬闯所有问题，而是要求它在被阻塞时立即停下并输出结构化信息。

典型阻塞场景：

- 缺少真实环境变量或密钥
- 外部服务未开通或账号未配置
- OAuth 之类流程必须人工授权
- 测试依赖真实账号、硬件或外部系统

阻塞时禁止：

- 提交 commit
- 将 `task.json` 标记为完成
- 假装任务已经交付

阻塞时必须：

- 在 `progress.txt` 写明已完成内容和阻塞原因
- 明确指出需要人工做什么
- 停止当前任务，等待人工解除阻塞

推荐阻塞模板：

```md
🚫 任务阻塞 - 需要人工介入

**当前任务**: 用户登录联调

**已完成的工作**:
- 登录页面已完成
- 前端认证流程已接入

**阻塞原因**:
- 缺少真实认证服务配置，无法完成联调测试

**需要人工帮助**:
1. 提供可用的环境变量
2. 配置认证服务回调地址

**解除阻塞后**:
- 运行 `npm run dev` 后继续验证
```

## 七、为什么这套模式有效

### 7.1 用文件系统承载上下文

AI 对话上下文会失真，但文件不会。  
`task.json` 和 `progress.txt` 把项目状态外置为稳定存储，使得 agent 会话可以随时重启、切换和接力。

### 7.2 用流程替代人工盯防

传统做法依赖人类不断审查 AI 的每一步。Dark Factory 改为让流程本身约束 AI：

- 不测试不能完成
- 不更新日志不能交接
- 不同步任务状态不能提交

### 7.3 用“小任务 + 强验证”换稳定性

模型在小任务上的成功率通常远高于大任务。  
因此这套模式要求任务必须可独立验证，并尽量缩小上下文和改动面。

### 7.4 用原子提交保证可回滚

每个任务对应一个完整 commit，有问题直接回退即可。  
这让自动化开发系统具备了类似流水线工位的可追溯性。

## 八、任务设计原则

如果要让 Dark Factory 真正跑起来，任务拆分质量比提示词花样更重要。

推荐遵循以下原则：

- 单任务目标明确，避免同时修改多个不相关模块
- 单任务可验证，最好有明确的测试手段
- 单任务可提交，避免半成品跨多个 session
- 单任务依赖尽量少，减少等待人工介入的概率

更适合的任务：

- “实现用户登录页并完成浏览器验证”
- “修复订单列表分页 bug，并补充测试”
- “新增设置页主题切换，并验证构建通过”

不适合的任务：

- “把整个平台重构一下”
- “完成支付、登录、部署和监控”
- “先随便做，后面再补测试”

## 九、落地模板

### 9.1 `task.json`

```json
{
  "tasks": [
    {
      "description": "实现用户登录页并接入认证流程",
      "passes": false
    },
    {
      "description": "新增个人中心页面并展示用户信息",
      "passes": false
    }
  ]
}
```

### 9.2 `progress.txt`

```md
## 2026-04-09 - Task: 实现用户登录页并接入认证流程
### What was done:
- 新增登录页面
- 接入认证 SDK
- 完成表单交互和错误提示

### Testing:
- npm run lint
- npm run build
- 浏览器手动验证登录流程

### Notes:
- 生产环境密钥仍需人工补充
```

### 9.3 Agent 操作规范

```md
1. 初始化环境
2. 读取 task.json，只选择一个未完成任务
3. 实现任务
4. 执行强制验证
5. 更新 progress.txt
6. 验证通过后再更新 task.json
7. 将代码、progress.txt、task.json 一并提交
8. 若阻塞，则记录阻塞并停止
```

## 十、适用边界

Dark Factory 很适合：

- AI Agent 持续开发
- 多会话接力开发
- 有清晰验收标准的 Web 和应用项目
- 可以通过脚本、构建、浏览器测试进行自动验证的任务

Dark Factory 不适合直接生搬硬套到所有场景：

- 需求高度模糊、无法拆分时
- 依赖大量线下决策或强业务判断时
- 无法构建自动验证闭环时
- 每个任务都需要高密度人工审阅时

## 十一、实践建议

如果要在真实项目中使用这套模式，建议优先做四件事：

1. 准备统一初始化脚本，如 `init.sh`
2. 为项目建立 `task.json` 和 `progress.txt`
3. 编写一份强约束的 agent 操作手册
4. 确保 `lint`、`build`、浏览器测试至少有两项可用

真正让 Dark Factory 成立的，不是“AI 很强”，而是“任务、验证、日志、提交”四个环节形成了闭环。

## 十二、一句话总结

Dark Factory 的本质不是“让 AI 自动写代码”，而是把软件开发组织成一套适合 AI 长时间稳定运行的生产系统。
